## What is it?

DuckLake is a lakehouse format — a way to store large analytical datasets as Parquet files on object storage (S3, Azure Blob Storage, GCS) while maintaining all table metadata (schemas, file locations, snapshots, transaction logs) in a standard relational SQL database rather than in scattered JSON/Avro metadata files on the object store itself. It was created by the DuckDB team and is accessed today via the `ducklake` DuckDB extension.

The core philosophy is elegantly simple: metadata management is a solved problem — relational databases with ACID transactions already do it perfectly. Rather than inventing a new file-based metadata format (like Iceberg's manifest files or Delta Lake's transaction logs), DuckLake stores everything in PostgreSQL, SQLite, or DuckDB. This delivers 10–100× faster metadata lookups (single indexed SQL query vs. multi-step object-store scans), ACID transactional guarantees across multi-table operations, schema evolution, time travel, and snapshotting without the operational burden of a dedicated catalog service.

v1.0, released today alongside DuckDB v1.5.2, is the first **production-ready, backward-compatible** release. It ships 108 merged PRs (68 focused on reliability/correctness), data inlining on by default, sorted tables, bucket partitioning (Iceberg-compatible), native `GEOMETRY` type support with spatial filter pushdown, a `VARIANT` type for semi-structured data, and experimental deletion vectors. There are already DuckLake clients for Apache Spark, Apache DataFusion, Trino, and Pandas, and MotherDuck offers a fully hosted DuckLake service.

## Why it matters (or doesn't)

The most immediately relevant intersection with Garrick's stack is **PostgreSQL + Azure**. CostEstDB already runs PostgreSQL on Azure with pgvector — and DuckLake's reference implementation treats PostgreSQL as a first-class catalog. Azure Blob Storage is an explicitly supported data store. In theory, Garrick could stand up a DuckLake over the existing Postgres instance with zero new catalog infrastructure. The "multiplayer DuckDB" scenario — multiple DuckDB instances coordinating concurrent reads and writes through a central PostgreSQL catalog — is a direct fit for a shared analytical layer that multiple team members or Azure Functions could query simultaneously.

The **`GEOMETRY` type** addition is worth flagging for GIS work. DuckLake v1.0 stores bounding-box statistics per Parquet file for geometry columns, enabling spatial filter pushdown at the file level. Municipal infrastructure datasets (water, sewer, stormwater geometries) stored in Azure Blob as Parquet could be queried with spatial predicates without scanning every file — an architecture that could complement ArcGIS workflows for large, infrequently-changing feature datasets.

That said, there's no immediate pain point DuckLake solves for Garrick's *current* projects. CostEstDB is an OLTP-adjacent workload (embedding lookups, cost record CRUD, pgvector similarity search) — not the large-scale analytical Parquet lakehouse DuckLake is designed for. The geometry and crash-report processing workflows are currently ArcGIS-native. DuckLake becomes compelling when Abonmarche starts accumulating analytical-scale datasets that outgrow a single Postgres table: multi-year infrastructure asset histories, aggregated cost estimating data across hundreds of projects, or large crash datasets that need to be shared across tools (DuckDB, Python, Spark).

## Technical details

- **License:** MIT — fully open, vendor-neutral spec
- **Current version:** 1.0 (production-ready, backward-compatibility guaranteed)
- **DuckDB requirement:** DuckDB v1.5.2+ (`INSTALL ducklake;`)
- **Catalog backends:** PostgreSQL, SQLite, DuckDB (local), MotherDuck (hosted)
- **Storage backends:** Azure Blob Storage, AWS S3, GCS, Cloudflare R2, local disk — Parquet only
- **Concurrency:** Multi-writer support when using PostgreSQL or SQLite catalog; single-writer for DuckDB-as-catalog
- **Key v1.0 features:** Data inlining (small inserts/updates/deletes stored in catalog DB, ≤10 rows default threshold — avoids Parquet proliferation), sorted tables (arbitrary SQL sort expressions for read optimization), bucket partitioning (Iceberg-compatible murmur3), `GEOMETRY` type with bbox stats, `VARIANT` type with shredded sub-field stats, deletion vectors (experimental, Puffin files)
- **Performance highlights:** `COUNT(*)` served from metadata stats (8–258× speedup on S3-backed tables); `~70×` faster `duckdb_views()` queries; top-N dynamic file pruning
- **Limitations:** No constraints, keys, or indexes (lakehouse paradigm); not suitable for OLTP point-update workloads; `GEOMETRY` spatial file-level pruning is bbox-only (full spatial pruning marked TODO); Variant inlining with Postgres catalog deferred to v1.1
- **Roadmap:** v1.1 targets Variant inlining for non-DuckDB catalogs, multi-deletion-vector Puffin files; v2.0 roadmap includes git-like branching, permission-based roles, and incremental materialized views
- **Adoption:** Already in production at dozens of companies; ranked in DuckDB's top-10 core extensions by download count; O'Reilly book in progress

## Recommendation

**Watch** — set a reminder to revisit in 1–2 months as Garrick evaluates what to do with growing analytical datasets. Specific triggers to upgrade this to `try_it`:

1. **CostEstDB historical data volume grows** past what's comfortable in a single Postgres table and needs a cheap Parquet-backed analytical layer with the existing Postgres instance as catalog.
2. **Infrastructure asset management** produces large geometry-heavy datasets that would benefit from spatial filter pushdown outside of ArcGIS.
3. **Multi-user DuckDB analytics** becomes a need — the PostgreSQL catalog "multiplayer" pattern solves DuckDB's single-writer limitation with zero new infrastructure for Abonmarche.

For now, the lowest-friction experiment would be: attach a test DuckLake to the existing CostEstDB Postgres instance with `DATA_PATH` pointing to Azure Blob Storage, load a copy of historical cost records as Parquet, and benchmark queries vs. direct Postgres. That's a 30-minute proof of concept that would answer whether DuckLake adds value to the existing stack.
